KessDB Whitepaper

Hyper-dense object-document storage for read-heavy edge software

KessDB is a proprietary, source-available proof-of-concept database engine that packs typed object documents into a single `.kess` file using Brotli-compressed chunks, source-generated codecs, and callback-scoped zero-copy reads.

Abstract

KessDB targets read-heavy applications that carry structured reference data, game definitions, rules, catalogs, pricing tables, or other mostly-local object graphs. Instead of deploying a database server or shipping loose JSON files, KessDB compiles data into a single `.kess` container optimized for compact storage, deterministic validation, and hardware-friendly reads.

The engine is built in C# for `net8.0` and `net10.0`, with a Native AOT posture: no runtime reflection mapping, no dynamic code generation, no expression tree compilation, and no SQL or ORM layer. Application documents are mapped by a source generator into explicit codecs and `readonly ref struct` views.

One file Header, manifest, schema, data chunks, optional index chunks, and WAL tail live in one `.kess` file.
64 KiB Default chunk target balances Brotli density with microsecond-scale lookup behavior.
0 alloc Steady-state callback reads avoid managed allocations after pools and caches are warmed.

Problem Space

Many applications use database infrastructure for data that behaves more like a compiled asset than a live transactional system. The cost is familiar: server processes, connection pools, serializers, object mappers, cache layers, operational scripts, and memory overhead. For edge nodes, games, serverless workers, and packaged reference datasets, that machinery can be larger than the data problem itself.

Plain compressed files reduce footprint but usually sacrifice random access. Traditional embedded relational databases provide excellent reliability and query capability but are not specialized for highly compressed object-document payloads or zero-copy span views. KessDB sits between those models: it keeps database-style durability and indexing primitives, but it stores data like a dense binary asset.

Architecture

The release-facing API is `KessDatabase` with typed `KessCollection<T>` instances. Every release `.kess` file uses a single container format, identified as `KessFormatVersion = 1`. Optional features such as indexes and richer field metadata are expressed through manifest counts and schema metadata rather than separate public file-format branches.

Document Mapping

Developers annotate normal partial POCOs or records with `[KessDocument]` and `[KessKey]`. The generator emits a codec, schema metadata, payload writer, materializer, index encoders, and a zero-copy view over `ReadOnlySpan<byte>`. Convention-based field ids are available for quick starts; stable schema names and explicit field ids are available for production files.

Chunk Layout

Base records are grouped into sorted Brotli-compressed chunks. Each decompressed chunk contains a record count, a fixed-width mini-index of primary keys, offsets, and lengths, and a trailing payload zone. This lets the engine binary-search within the chunk instead of scanning or materializing all records.

Secondary Indexes

Generated indexes are collection-scoped and AOT-safe. They support single-field indexes, composite tuple indexes, unique constraints, string prefix lookups, and numeric range lookups. Persisted Brotli-compressed index chunks hydrate immutable in-memory index snapshots at open time.

Zero-Copy Read Path

The hot path is callback-scoped. A caller asks for a record through `WithView`, `WithMany`, `WithRange`, or generated index helpers. The engine acquires an immutable read state, resolves the key or index result, reads with offset-based file IO, and invokes the callback with the exact record slice.

powers.WithView(42, static record =>
{
    PowerDocumentView view = new(record);
    int power = view.Power;
    ReadOnlySpan<byte> nameUtf8 = view.NameUtf8;
});

The record span cannot escape the callback. This protects pooled memory lifetime while allowing generated `readonly ref struct` views to expose UTF-8 strings, binary fields, nullable fields, arrays, enums, and common CLR scalars without materializing managed objects.

Convenience materializers such as `Load`, `LoadMany`, and `LoadRange` are also generated. They allocate normal classes, strings, arrays, and lists by design and are not part of the zero-allocation read guarantee.

Transactions And Durability

KessDB uses a single-writer, multi-reader model. Writers serialize through a write gate and append committed changes to the WAL. Readers use immutable read states and merge committed WAL overlay entries over base chunks. This provides read-committed behavior without MVCC, background server processes, or distributed coordination.

Immediate mutations use automatic transactions. Explicit `KessTransaction` tokens batch writes and flush at `Commit(transaction)`. The release durability contract is commit-boundary durability, not a hardware flush after every individual WAL append inside an explicit transaction.

Compaction rewrites visible base records and committed WAL overlays into fresh Brotli data and index chunks, then clears the WAL tail. Auto-compaction is opt-in for density-focused workflows because compaction can add commit latency.

Proof And Diagnostics

KessDB includes first-party proof and diagnostic tooling through `KessDB.Tool`. The CLI validates files, reports segment accounting, compacts databases, runs benchmark smoke profiles, salvages readable raw payloads from damaged files, and exercises release-candidate workloads.

Command Purpose
kess validate Validate header, manifest, schema, chunk tables, CRCs, mini-indexes, index chunks, and WAL recovery status.
kess stats Report file-size accounting, chunk counts, WAL bytes, index overhead, and compression ratios.
kess benchmark-smoke --canonical-v0.1 Generate deterministic release proof data for size, point reads, batch reads, range reads, cache reads, WAL overlay reads, and index reads.
kess proof-rc Exercise concurrent reads, commits, deletes, compaction, reopen recovery, validation, uniqueness checks, rich generated documents, and salvage on corrupt copies.

Compression claims are currently worded as "up to 96.5%" unless the checked-in canonical benchmark report demonstrates that exact threshold. Current release gates favor honest repeatability over fragile headline metrics.

Boundaries And Non-Goals

KessDB is not intended to replace general-purpose database systems. It has no SQL layer, LINQ query planner, distributed transactions, server daemon, replication protocol, or MVCC snapshot isolation. Its target is local, read-heavy, schema-owned object data where compact packaging and low-allocation reads matter more than ad hoc query flexibility.

Secondary indexes are optimized for read-heavy datasets and moderate update rates, not OLTP-style mutable B-tree workloads. Commit publication rebuilds immutable snapshots so reads remain writer-lock-free.

Integrity is not security. CRC validation detects accidental corruption, torn writes, and malformed files. It is not tamper resistance, encryption, signing, authenticity, or a cryptographic file-level integrity guarantee.
